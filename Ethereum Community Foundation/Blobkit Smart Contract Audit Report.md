# **Review Summary**

The Auditware team has performed a security review on BlobKit, a protocol enabling  
trustless blob data storage on Ethereum using EIP-4844 blob transactions.

The protocol solves the fundamental challenge that browser wallets cannot execute  
blob transactions directly by implementing a trust-minimized proxy architecture with  
smart contract escrow for payments.

The scope consisted of the escrow contract:

* [BlobKitEscrow.sol](https://github.com/ETHCF/blobkit/blob/5f3d4098fe68bdb0eb6e464363286bbaef9bdb73/packages/contracts/src/BlobKitEscrow.sol)

Audit commit hash: [5f3d409](https://github.com/ETHCF/blobkit/commit/5f3d4098fe68bdb0eb6e464363286bbaef9bdb73) 

## Contract Overview

### Core Components:

* **Escrow Contract** \- Trustless payment coordination between users and proxy operators.  
* **Typescript SDK** \- Universal interface for blob operations across browser and Node.js environments.  
* **Proxy Server** – Execute blob transactions on behalf of browser users

### Example Operation Sequence:

* User calls **depositForJob(jobId)**, with a unique ID generated by the SDK  
* On a browser use case, the SDK sends blob data to the proxy server  
* The proxy then executes the blob transaction and generates a ECDSA signature as a submission proof  
* Proxy calls **completeJob(jobId, blobTxHash, proof)**  
* Escrow contract verifies proof signature  
* Full payment is being transferred to the proxy  
* The job is marked as complete for replay protection


## Security Overview

### Access Model:

**User Privileges:**

* Job creation with payment  
* Refund claiming after timeout

**Proxy Privileges:**

* Job completion (with valid proof)  
* Self fee configuration (0-10%)

**Owner Privileges**:

* Proxy authorization/deauthorization  
* Timeout configuration (1 minute to 24 hours)  
* Emergency pause/unpause  
* Emergency withdrawal (when paused)

### Emergency Response:

* Owner can pause contract to stop new deposits and completions  
* Existing jobs remain refundable during pause  
* Emergency withdrawal available during pause

### Security Assumptions:

* **OpenZeppelin Libraries**: Trusted and battle-tested implementations  
* **Ethereum Consensus**: Block timestamps cannot be significantly manipulated  
* **EVM Security**: ECDSA and keccak256 are cryptographically secure  
* **Owner Integrity**: Contract owner acts in good faith  
* **Proxy Reputation**: Authorized proxies have economic incentive to complete jobs

As with all smart contracts, it is highly recommended to:

* Conduct regular and thorough audits to identify and mitigate vulnerabilities.  
* Achieve the highest possible test coverage, ideally approaching 100%.  
* Maintain comprehensive and up-to-date documentation.  
* Participate in auditing contests and/or a bug bounty program.  
* Implement on-chain monitoring for real-time threat detection and mitigation.

Protocol-specific monitoring recommendations:

* Job Completion Rates: Unusual completion patterns may indicate attacks  
* Large Deposits: Monitor for unusually large job amounts  
* Proxy Authorization Changes: Alert on new proxy authorizations  
* Failed Transactions: High failure rates may indicate attack attempts  
* Contract Balance: Unexpected balance changes warrant investigation

# **AW-M-01: Unlimited Owner Emergency Withdrawal** {#aw-m-01:-unlimited-owner-emergency-withdrawal}

**Severity:** *Medium*							**Status:** *Unmitigated*

**Locations:**

* [packages/contracts/src/BlobKitEscrow.sol\#:L260-L264](https://github.com/ETHCF/blobkit/blob/5f3d4098fe68bdb0eb6e464363286bbaef9bdb73/packages/contracts/src/BlobKitEscrow.sol#:L260-L264)  
* [packages/contracts/src/BlobKitEscrow.sol\#:L245-L247](https://github.com/ETHCF/blobkit/blob/5f3d4098fe68bdb0eb6e464363286bbaef9bdb73/packages/contracts/src/BlobKitEscrow.sol#:L245-L247) 

**Description:**  
The BlobKitEscrow contract implements an emergency withdrawal mechanism that grants the contract owner absolute authority to seize all user funds without consent, time delays, or external oversight.

This mechanism operates through a two-step process:

* Pausing the contract to halt normal operations  
* Immediately withdrawing the entire contract balance to the owner's address.

This design creates a critical centralization risk where a single private key controls the fate of all user deposits.

The emergency powers are intended for crisis situations but lack safeguards against abuse, creating conditions where users must place unlimited trust in the owner's benevolence and key security practices.

**Recommendation:**

* Implement time-locked withdrawals: Add minimum 7-day delay between pause and emergency withdrawal.  
* Add multi-signature requirements \- require 3-of-5 signatures for emergency functions.  
* Enable proportional user withdrawals \- allow users to withdraw their specific deposits during emergencies.  
* Create governance oversight \- transition emergency controls to decentralized governance with transparent voting.

# **AW-L-01: Zero Blob Transaction Hash Acceptance** {#aw-l-01:-zero-blob-transaction-hash-acceptance}

**Severity:** *Low*								**Status:** *Unmitigated*

**Locations:**

* [packages/contracts/src/BlobKitEscrow.sol\#L139-L163](https://github.com/ETHCF/blobkit/blob/5f3d4098fe68bdb0eb6e464363286bbaef9bdb73/packages/contracts/src/BlobKitEscrow.sol#L139-L163) 

**Description:**  
The **completeJob()** function in **BlobKitEscrow** contract accepts **bytes32(0)** as a valid blob transaction hash without performing input validation.

This allows authorized proxies to mark jobs as completed while providing meaningless blob references, undermining the protocol's core purpose of facilitating verifiable blob storage on Ethereum.

The contract then stores the zero hash in the job record and emits completion events, creating false completion signals for external systems that expect valid transaction hashes for blob data access.

This will cause anyone who tries to access blob data via zero transaction hash to fail, causing integration errors and undermining data integrity expectations.

**Recommendation:**

* Add the missing input validation in **completeJob()** function, like so:  
  function completeJob(bytes32 jobId, bytes32 blobTxHash, bytes calldata proof) external nonReentrant whenNotPaused {  
     if (\!authorizedProxies\[msg.sender\]) revert UnauthorizedProxy();  
     if (blobTxHash \== bytes32(0)) revert InvalidBlobTxHash();  
      
     Job storage job \= jobs\[jobId\];  
     // ...  
  }  
    
  


# **AW-L-02: Fee Calculation Inconsistency Causing Integration Confusion** {#aw-l-02:-fee-calculation-inconsistency-causing-integration-confusion}

**Severity:** *Low*								**Status:** *Unmitigated*

**Locations:**

* [packages/contracts/src/BlobKitEscrow.sol\#L160-L165](https://github.com/ETHCF/blobkit/blob/5f3d4098fe68bdb0eb6e464363286bbaef9bdb73/packages/contracts/src/BlobKitEscrow.sol#L160-L165) 

**Description:**  
The BlobKitEscrow contract contains inconsistency between its economic model implementation and its external interface signals.

The contract calculates proxy fees using percentage-based logic, suggesting a fee-deduction model where proxies receive a percentage fee and users retain the remainder for blob storage costs.

However, the actual implementation transfers the entire user deposit to proxies, operating as a service-payment model where proxies cover all costs.

This ambiguity creates multiple risks:

* External systems may incorrectly interpret the payment structure  
* Users may have false expectations about cost allocation  
* Future integration partners may implement incorrect financial logic based on misleading event emissions

**Recommendation:**

* Remove misleading fee calculation and update event to reflect actual payment model:  
  // Remove confusing fee calculation in completeJob function  
    
  // uint256 proxyFeePercent \= proxyFees\[msg.sender\];  
  // uint256 proxyFee \= (job.amount \* proxyFeePercent) / 100;  
    
  // Update event emission to reflect actual amount transferred:  
  emit JobCompleted(jobId, blobTxHash, job.amount);  
    
* Or implement the actual fee deduction if fee-based model intended:  
  // Calculate actual fee deduction  
  uint256 proxyFeePercent \= proxyFees\[msg.sender\];  
  uint256 proxyFee \= (job.amount \* proxyFeePercent) / 100;  
  uint256 serviceCost \= job.amount \- proxyFee;  
    
  // Transfer only service cost to proxy, keep fee separate  
  (bool success,) \= payable(msg.sender).call{value: serviceCost}("");
